Data Collection Template

Guide on Data Set Completion

Created by Turlough McAnespy

Document Purpose

The Data Set Collection document is intended for communicating the required Data Set for the Registry or Study being developed, and to act as a source of truth that the development can be tested against.

As such it needs to be completed with rigour, to avoid ambiguity (as a collaboration between Client & OpenApp) & be kept up to data.

Spreadsheets

  • Each of the sheets represents the data to be gathered by a particular form e.g. Enrolment form, Encounter form, End of Data Collection form etc
  • Sheets can be added to accommodate other data gathering forms
  • Sheets can be added to help manage communication, but these should be deleted as soon as they are no longer relevant

Spreadsheets

  • The form spreadsheets to be completed by the study manager are highlighted with green tabs
  • Spreadsheets for internal OpenApp use are highlighted with blue tabs,
  • while the help/reference spreadsheet has a white tab

Information style

  • The labels, values and help text should be written in Arial (or a similar Sans Serif font)
  • All instructions should be written in a format to clearly differentiate it from the text above - if not in a column of instruction then as Italic or Time New Roman

Columns and Rows

  • Each row on the spreadsheet is used to define an individual item of data (field) stored within the Registry or Study.
  • The columns indicate the details for each field. The 'comment' attached to the heading of each column provides a brief explanation of it's use.
  • As some Data sets are more straight forward than others some of the columns are hidden by default and will only be displayed if it is deemed useful to do so

Columns explained

There are 5 general types of columns in Template;

  • Form Information Hierarchy - generally dealing with the field groupings
  • Field attributes - the specific details for each field such as label, UI element type, constraints etc.
  • the term 'Business Rules' is usually used while discussing the following
    • Display logic - the rules that govern when the fields are displayed
    • Validation - the rules that govern whether a field is in a valid state for saving the form
    • Verification - the rules that govern if a Automated Query is triggered on a field value that has been saved
  • Discussion - columns for supporting the discussion of the Dataset between the Client & OpenApp
  • Agreement confirmation - 2 columns to allow us to track which fields can be put into a development Sprint for delivering

Order of completion

The form is typically created in 2 Sprint stages - first the fields are created and then rules governing Dispay logic, Validation and Verification are added. It makes most sense to complete the columns regarding Hierarchy and Field attributes first, as the completion of the rest of the columns are dependant on these being correct.

Form Information Hierarchy

There are 4 potential levels to the Form information hierarchy, only the first 2 are displayed by default;

  1. Section -
  2. Sub-Section
  3. Sub-Sub-Section
  4. Sub-Sub-Sub-Section

Section

A form will frequently have sections containing groups datafields from similar thematic areas - for example - Spirometry, Lab Tests, Radiology etc.

There is a Section navigation menu displayed on the left-hand side of a form.

If any of the Sections are optional, a User can select these Sections by clicking on the checkbox to the left Section name

Sub-Section

Where a Section is very large (for example, it is too large to fit entirely on a laptop screen), it should be broken down into further Subsections

Sub-Sub-Section and Sub-Sub-Sub-Section

Where they bring value in communicating the Form hierarchy it might be useful to also create further hierarchical groupings Sub-Sub-Sections and/or Sub-Sub-Sub-Sections. These should only be used when they deliver real value

Field Attributes

Each field has a set of attributes. Some of these are not relevant to every field but some of the most important include;

  • Field Label - the text that will appear on screen and identifies the field
  • Control - used to define the UI mechanism used to interact with the Field (ex: Text Box, Rich Text Box, Calendar, Radio Button, Check Box, Dropdown, Combo Box etc.)
  • Options - these are the options available to the User when the Control type contains multiple options as with a Radio button, Checkbox, Dropdown menu etc.
  • Autogenerated - it is important to know if a field is to be entered manually or is generated by calculating other entered values. A definition of how this is calculated needs to be defined in the xxx column
  • Float - if a field is to record a floating-point (decimal) number then then number of decimal places should be recorded here with X’s used to indicate how many numbers are required. For instance XXX.XX would allow 3 numbers before the decimal point and 2 numbers after. It will not allow values greater than 999.
  • Units - it is important to state the Unit of measurement explicitly to create reliable Data
  • Ongoing - if this field needs to be copied over from one Encounter to the next then enter Yes in this column
  • Repeating Field - if this field will need to be filled in multiple times then it should be marked as a Repeating Field. Examples include Treatments, Hospitalisations etc.
  • Help Text - if a field needs explanatory text on screen, the text can be added in this column. Column is hidden by default
