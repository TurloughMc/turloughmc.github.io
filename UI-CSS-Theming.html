Methods for Theming

Ideas around which techniques to use and when

Created by Turlough McAnespy

Advice on Theming

  • Avoid if possible - make sure there is a real business case
  • KISS - always reduce the complexity involved
  • Only make changes to cosmetic aspects - avoiid altering box-model
  • Enforce rules - deviation is expensive
  • Use it as an up-sell - 'Any deviation is going to cost'

So why would we theme

There are only two main reasons


  1. Branding
  2. Accessibility

Methods for Theming

4 Techniques with different scenarios

#1 - Theme Layer

Overiding default style with additional CSS

  1. Most common approach
  2. Start off with the base/default styling
  3. Use additional, subsequent CSS to override and redefine

Suited to;

  1. Bespoke theming for clients.

The Good;

  1. You can swap layers out
  2. You can theme anything

The Bad;

  1. Lots of redundancy
  2. Wasted CSS over the wire
  3. Shards styles across multiple files

#2 - Stateful Theming

Styling a UI based on a state or condition

  1. Different skins based on conditions
  2. Allow themes to be switched on the client
Suited to;
  1. Style switchers
  2. Individually styled sections or areas of a site
Stateful Utility Classes; A utility class that tracks the current theme The Good;
  1. Bundle a number of themes into one codebase
  2. Great for style-switchers
  3. Perfect for 'areas' of a site
  4. You can theme anything
The Bad;
  1. Bundle a number of themes into one codebase
  2. Potential redundancy

#3 Config Theming

Invoking a theme based on settings

  1. Switching between themes dev-side
  2. Compiling the same CSS with a different skin based on a setting
Suited to;
  1. Spitting out differently themed variants of a UI
  2. Offering off-the-shelf themes to clients
The Good;
  1. Only send as little CSS over the wire as necessary
  2. Alter the entire theme fom one location
  3. You can theme anything
The Bad;
  1. Lots of logic in your Sass
  2. Only supports a finite number of themes
  3. Theme information is spread out through the codebase
  4. Adding themes could be laborious

#4 Theme Palettes

Holding entire themes in a palette file

  1. Preload all of your theme settings into a palette
  2. Inject these settings into the project later
Suited to;
  1. Fully bespoke theming for clients
  2. Adding specific branding to a project
The Good;
  1. Zero redundancy
  2. Perfect for bespoke theming - just plug in the client's brand colours
  3. Completely restyle the project from one file
The Bad;
  1. Limited mainly just to theming colours

When to use

  • Do the themes need to change once they're on the client? Yes for Accessibility changes = Stateful
  • Do you have themes that you want people to be able to toggle? = Stateful
  • Do you have sections of the site that need to look different? = Stateful
  • Do you have preset themes that a client can choose from? Config
  • Do you offer bespoke theming for white-label clients? - Theme Layer or Theme Palette
