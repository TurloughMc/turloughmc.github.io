What a Design System?

A Design System is the single source of truth which groups all the elements that will allow the teams to design, realize and develop a product.

Why is this useful?

There are a number of different areas needing needing designed and documented within any digital product. These include;

  • Functional User Experience Patterns - the essence of which is behavioural
  • Perceptual Patterns - iconography styles, colors and typography
  • User Interface Components - tangible building blocks of the interface, such as a button, a header, a form element, a menu

Why a Design System?

The "things" being documented are very different in their nature and hence benefit from being documented in differing ways

  • Functional User Experience Patterns - Behaviour
  • Perceptual Patterns - Style
  • User Interface Components - Code + Style

Why a Design System?

Without a coherent strategy for where each piece of documentation 'lives', it quickly becomes un-manageable - patchy, featuring duplication & requiring too much effort for proper engagement from other stakeholders

Potential Solution

Create a Design System

Design System Process

Design System Characteristic

  1. Input from more than one viewpoint
  2. Space for Diverge > Converge on Problem & Solution
  3. Creation of source of truth
  4. Documentation of the Solution
  5. Developer has Solution to use out of the box and can concentrate on other problems

How

  1. Build on top of the toolkit we already use (Bootstrap + our own bits), modifying and adding to as needed
  2. We don't need to break anything that is already working ok
  3. We don't need to forget and re-learn what is already understood reasonably well
  4. Refine and document legacy
  5. Add new documentation of Components/Patterns as these are developed

Where to start creating tangibles?

  1. Users glossary list - Google Doc. See .odt and .xmind (particularly doc-tree)
  2. Component Inventory list - including up to the size of Page - Google Doc
  3. Crystalize Pages identified above and make them the defaults out of the box on install
  4. Fuctionality Epics list - Google Doc
  5. Styleguide - in what?

Why an Openapp Toolkit?

Because we can address many of the previous issues in 1 go, including;

  1. Supporting Designers Protoyping
  2. Fast tracking work from Figma prototypes to working code
  3. CSS - Tokens are mirrored between Figma Comnponents & SCSS Variables
  4. Clean out the technical debt plaguing our current approach
  5. Document design so system is maintainable

To support the Big Picture

Project roadmap

  1. Identify generic Pattern areas, and list of functional stories within these, referencing UX Pattern libraries as we go - create an document, ...
  2. UX - Create the Figma Components to support these where BS doesn't supply them
  3. UI - Create BootstrapVue Components where the above don't already exist
  4. API Work so the Toolkit interfaces with the backend
  5. Internal - Sprint to tie it all together as a generic functioning system and do some aesthetic work so it doesn't look 'under designed'
  6. External - Repeat process again, this time at a more specific level to include what might only relate to each customer project
